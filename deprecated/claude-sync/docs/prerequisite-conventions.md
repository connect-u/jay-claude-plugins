# Prerequisite Conventions: claude-sync를 쓰기 위한 팀 합의

> **이 문서의 위치:** claude-sync는 *도구* 다. 도구가 잘 작동하려면 *합의된 규약* 위에서 동작해야 한다. 이 문서는 그 합의 항목들의 체크리스트다.
>
> **강도:**
> - 🔴 **필수** — 이게 없으면 도구가 사실상 작동 안 함
> - 🟡 **권장** — 없으면 도구 효과가 크게 줄어듦
> - 🟢 **선택** — 팀 상황에 따라

---

## 0. 왜 규약이 필요한가

claude-sync는 "합의된 것을 *기록하고 강제하는* 도구" 다. 합의 자체가 없으면 도구는 빈 그릇이다.

예를 들어:
- 브랜치 prefix 정책이 없으면 → Sandbox/Production 모드 자동 전환이 작동 안 함
- ADR을 어디까지 쓸지 합의 없으면 → 노이즈 폭발 또는 묻힘
- 슬랙 채널이 정해지지 않으면 → Coordination 봇이 어디에 멘션할지 모름

**도구 도입 전 30분~1시간의 팀 합의가 도구 효과의 80%를 결정한다.**

---

## 1. 브랜치 전략 🔴 필수

### 1.1 Prefix 규칙

claude-sync는 브랜치 prefix로 모드를 자동 전환한다. 따라서 prefix 사용이 강제된다.

**최소 합의:**
- `feature/*` — 정규 기능 개발 (Production 모드)
- `fix/*` — 버그 수정 (Production 모드)
- `spike/*` 또는 `experiment/*` — 실험 (Sandbox 모드)
- `hotfix/*` — 긴급 수정 (긴급 모드 진입 신호)

**선택:**
- `chore/*`, `refactor/*`, `docs/*` 등

### 1.2 브랜치 수명

🟡 트렁크 기반 개발 권장. 브랜치는 길어야 1~2일.

이유: 장기 브랜치는 의미적 충돌의 온상. claude-sync도 도울 수 있지만, 짧은 브랜치 자체가 가장 좋은 충돌 방지책.

기능이 크면 기능 플래그로 감추고 작은 단위로 자주 머지.

### 1.3 메인 브랜치 정책

🔴 `main` 또는 `master`가 모든 `.sync/` 객체의 source of truth.

## 2. ADR 작성 정책 🔴 필수

### 2.1 ADR 작성 기준

**무엇을 ADR로 남길지 합의해야 한다.** 이게 없으면 노이즈 폭발 또는 누락.

**기본 권장 (조정 가능):**

ADR로 남긴다:
- 아키텍처 패턴 선택 (어떤 큐, 어떤 캐시 전략 등)
- 외부 의존성 추가/변경
- API 계약 / 데이터 스키마
- 정책 결정 (에러 처리, 보안, 데이터 처리 방식)
- 도메인 핵심 용어 정의

ADR로 안 남긴다:
- 변수명/함수명 (코드 리뷰에서 다룸)
- 작은 리팩토링
- 명백한 버그 수정 방향
- 라이브러리 minor 버전 업

**팀이 자체 조정해야 할 영역:**
- 라이브러리 선택 (어떤 팀은 ADR, 어떤 팀은 그냥)
- 테스트 전략 (한 번 정하면 그걸로, 매번 ADR X)

### 2.2 ADR 형식

🔴 합의된 한 가지 포맷.

기본 권장 — Michael Nygard 형식 (`status`, `context`, `decision`, `consequences`).
claude-sync는 여기에 v2 확장 (`trigger_conditions`, `scope_limits`, `maturity_score`) 추가.

### 2.3 ADR 번호 정책

🔴 단일 증가 번호 (ADR-0001, ADR-0002, ...). 영역별 prefix X (혼란만 야기).

### 2.4 ADR 변경 정책

🟡 머지된 ADR은 직접 수정 금지. 변경 시:
- Superseded: 새 ADR이 대체
- Scope-Limited: 일부 영역만 적용
- Reverted: 완전 철회

## 3. Intent 등록 정책 🔴 필수

### 3.1 언제 등록하나

🔴 새 브랜치를 만들고 *처음 작업 시작 전*에 `/sync claim`.

🟡 등록 안 한 브랜치는 머지 차단 (옵션, 팀 결정).

### 3.2 어떤 작업이 Intent 대상인가

🔴 다음은 반드시 Intent:
- 새 기능 개발
- 인터페이스(함수 시그니처, API, 스키마) 변경
- 공유 코드(`/shared/*`, `/common/*`) 수정

🟢 다음은 선택:
- 단순 버그 수정
- 문서 변경
- 테스트 추가

### 3.3 Intent 갱신 의무

🟡 작업하면서 *예상보다 영역이 확장*되면 Intent 갱신.

claude-sync가 자동 감지해서 갱신 제안하지만, 사용자가 명시적으로 정정할 수도 있어야 함.

---

## 4. 슬랙 / 메신저 정책 🔴 필수 (Active Coordination 사용 시)

### 4.1 Coordination 채널

🔴 봇이 멘션할 채널 지정.

권장:
- `#dev` 또는 `#engineering` — 일반 조율
- `#dev-urgent` — 긴급 조율 (선택)
- `#dev-decisions` — 합의 결과만 모이는 채널 (선택, 추적용)

### 4.2 합의 확정 신호

🔴 명시적 신호로만 합의 인정.

권장:
- ✅ react — 동의
- ❌ react — 명시적 거부
- 💬 thread reply — 토론

자연어 ("ok", "응", "좋아")는 합의로 인정하지 않음. 봇이 모호하면 "✅ 부탁드려요" 라고 다시 요청.

### 4.3 비개발자 참여

🟡 PM, 디자이너, 보안 담당자가 합의에 참여할 수 있는가?

만약 참여한다면:
- 그들의 슬랙 계정과 역할을 claude-sync에 등록
- 봇이 그들에게 멘션할 때 친절한 안내 ("✅ 또는 ok 라고 답해주세요")

### 4.4 멘션 강도

🟢 Critical 멘션 vs FYI 멘션 정책.

권장:
- Critical: @mention + DM
- FYI: 채널 메시지만, mention X

---

## 5. 역할 정의 🟡 권장

### 5.1 최소 역할

각 영역에 책임자가 있어야 도구가 누구를 부를지 안다.

권장 역할:
- **Tech Lead** — 큰 결정의 최종 승인자
- **Security Reviewer** — 보안 정책 (PII, 인증 등) 영향 시 자동 멘션 대상
- **Infra Owner** — 인프라 변경 시 자동 멘션 대상
- **Domain Owners** — 도메인별 (결제, 알림 등) 책임자

### 5.2 역할 등록

claude-sync 설정 파일 (`.sync/team.yaml`) 에 명시:

```yaml
team:
  - { slack: "@alice", role: ["tech_lead", "domain:payments"] }
  - { slack: "@bob",   role: ["domain:notifications"] }
  - { slack: "@diana", role: ["infra_owner", "security_reviewer"] }
```

## 6. PROJECT.md 작성 🔴 필수

### 6.1 분량

🔴 A4 1~2장 이내. 길면 안 읽힘.

### 6.2 포함 항목

```markdown
# Project: [이름]

## 한 줄 정의
[X를 위한 Y를 만든다]

## 유저 시나리오 (3~5개)
1. ...

## 시스템 경계
- 우리가 만드는 것:
- 외부 의존:

## 비기능 요구사항 (아키텍처 영향)
- ...

## 기술 스택 (바꾸기 어려운 결정)
- 언어:
- 주요 프레임워크:
- DB:

## 팀 (옵션)
- (이 파일에 두거나 .sync/team.yaml에)
```

### 6.3 변경 정책

🔴 PROJECT.md 수정은 Heavy 모드. 팀 전원 합의.

---

## 7. 디렉토리 구조 합의 🟡 권장

### 7.1 영역 분리

claude-sync는 영역 충돌을 감지한다. 영역이 명확해야 작동.

**최소 합의:**
- 각 도메인의 디렉토리 경로 (`/payments/*`, `/notifications/*` 등)
- 공유 코드 위치 (`/shared/*`, `/common/*`, `/lib/*` 등)
- 인프라/플랫폼 코드 위치 (`/platform/*` 등)

### 7.2 공유 코드 승격 정책

🟡 어떤 조건에서 코드를 `/shared/`로 승격하는가?

권장:
- 2명 이상이 같은 패턴 구현 시 → Cross-pollination 검사 트리거
- 승격 결정은 Coordination 으로 처리

---

## 8. 테스트 정책 🟢 선택 (도구 기능 아님, 일반 권장)

claude-sync 자체와 직접 관련 없지만, 협업 효과 극대화를 위해 권장.

- 머지 전 테스트 통과 의무
- 인터페이스 변경 시 호출자 테스트 동시 수정
- 신규 ADR에 따른 기존 코드 마이그레이션이면 테스트 우선

---

## 9. PR 정책 🟡 권장

### 9.1 PR 크기

🟡 작게. 500 LOC 넘으면 분할 권장.

### 9.2 PR 설명에 포함할 것

claude-sync가 자동 생성 도움:
- 관련 Intent ID
- 관련 ADR ID (새로 만들거나 참조)
- 관련 Coordination ID (있다면)
- 영향 영역 (자동 감지)

### 9.3 머지 정책

🟢 선택:
- ADR 위배 감지 시 머지 차단
- 미확정 Coordination 관련 PR은 머지 차단
- 영향 가능 영역 표시된 PR은 영향받는 사람 리뷰 요구

---

## 10. 데이터 / 프라이버시 정책 🔴 필수

### 10.1 .sync/drafts/

🔴 대화 발췌가 저장됨. 민감 정보 포함 가능.

권장:
- `.gitignore` 또는 별도 저장소 검토
- 또는 redaction 정책 (PII, 비밀번호, 토큰 자동 마스킹)
- draft 보관 기간 (예: 30일 후 자동 삭제)

### 10.2 외부 통합

🔴 슬랙으로 전송되는 컨텍스트에 민감 정보 없게.

- 코드 직접 인용 X (요약만)
- 토큰/시크릿 자동 감지 후 거부

## 11. 합의 체크리스트

도구 도입 전 30분~1시간 회의에서 이걸 채워라.

```markdown
# Team Conventions Checklist

## 브랜치
- [ ] feature/* 사용?
- [ ] spike/* 또는 experiment/* 사용?
- [ ] hotfix/* 사용?
- [ ] 메인 브랜치 이름: ___

## ADR
- [ ] ADR로 남길 카테고리:
  - [ ] 아키텍처 패턴
  - [ ] 외부 의존성
  - [ ] API/스키마
  - [ ] 정책
  - [ ] 도메인 용어
  - [ ] 기타: ___
- [ ] 형식: Nygard / 자체정의 (___)
- [ ] 번호: 단일증가 / 영역prefix

## Intent
- [ ] /sync claim 의무 대상 합의됨
- [ ] 등록 안 한 브랜치 머지 차단? Y/N

## 슬랙
- [ ] Coordination 채널: ___
- [ ] 합의 신호: ✅ react / 기타 ___
- [ ] 비개발자 참여: Y/N
  - 참여 시 역할: ___

## 역할
- [ ] Tech Lead: ___
- [ ] Security Reviewer: ___
- [ ] Infra Owner: ___
- [ ] Domain Owners: ___

## PROJECT.md
- [ ] 초안 작성됨
- [ ] 팀 전원 검토함

## 디렉토리
- [ ] 도메인 경로 합의됨
- [ ] 공유 코드 경로: ___
- [ ] 승격 정책: ___

## PR
- [ ] 크기 권장: ___
- [ ] ADR 위배 머지 차단? Y/N

## 프라이버시
- [ ] .sync/drafts 보관 정책: ___
- [ ] redaction 룰: ___
```

## 12. 회의 진행 가이드 (도구 도입 첫 회의)

권장 진행:

**1) 미션 정렬 (5분)**
- 왜 이 도구를 도입하나? 어떤 문제를 풀려고 하나?

**2) PROJECT.md 초안 (15~20분)**
- 한 줄 정의부터 채우기
- 기술 스택 합의

**3) 브랜치/ADR/Intent 합의 (10~15분)**
- 위 체크리스트 따라가기

**4) 슬랙/역할 합의 (5~10분)**

**5) 첫 ADR 작성 (5분)**
- "ADR-0001: claude-sync 도입 결정" 자체를 ADR로 남김
- 이 회의에서 합의한 내용 요약

**총 1시간.**

## 13. 자주 묻는 질문

### Q. 기존 프로젝트에도 적용 가능한가?
A. 가능. 단, 기존 코드에 ADR 소급 적용은 X. 새 결정부터 시작.

### Q. 1인 개발에도 유용한가?
A. 부분적으로. Coordination 같은 다자 기능은 무용. 하지만 결정 캡처 / Reversal 감지는 1인에게도 가치 있음 ("3개월 전 내가 왜 이렇게 정했지?" 문제 해결).

### Q. 모든 합의를 한 번에 완벽히 해야 하나?
A. 아니. 최소 합의 (브랜치 + ADR 기본 + 슬랙 채널) 만으로 시작. 나머지는 도구 쓰면서 진화.

### Q. 합의를 어겼을 때 어떻게 되나?
A. 도구가 경고하지만 강제하지 않음. 강제할지는 팀 정책 (PR 차단 등).
