# ADR-0001: claude-sync 자체에 claude-sync 철학 적용 (도그푸딩)

- **Status:** Accepted
- **Date:** 2026-05-13
- **Author:** jay
- **Confirmed by:** [jay]

## Context

claude-sync는 LLM 협업 도구다. 우리가 만드는 도구가 진짜로 쓸 만한지 검증하려면, 우리 자신이 이 도구의 철학을 따르며 개발해야 한다.

이 결정은 의도적이다 — 도구를 만드는 사람이 그 도구의 가장 가혹한 사용자가 된다.

## Decision

claude-sync 프로젝트는 자체 개발 과정에서 다음을 따른다:

1. **PROJECT.md** — Frozen Core 유지. 변경 시 ADR.
2. **`.sync/decisions/`** — 모든 주요 결정을 ADR로.
3. **`.sync/intents/`** — 새 작업 시작 시 (수동이라도) intent 파일 생성.
4. **`.sync/coordinations/`** — 외부 의견 수렴이 발생하면 기록.
5. **`.sync/drafts/`** — 결정 후보를 모아두는 그릇.

도구가 아직 없으므로 *수동으로* 한다. 이 수동 과정에서 발생하는 마찰이 곧 도구가 자동화해야 할 지점이다.

## Consequences

**긍정적:**
- 도구 미존재 시점에 워크플로우의 마찰점을 발견할 수 있음
- 실제 사용 경험이 design.md 의 개선 driver가 됨
- 외부 사용자에게 "우리도 쓴다" 라고 말할 수 있는 신뢰성

**부정적:**
- 수동 작업의 부담 — 매번 ADR 작성, intent 파일 생성
- 1인 프로젝트에선 Coordination 같은 다자 기능 검증 불가

**리스크 완화:**
- 수동이 너무 부담스러우면 그 자체가 데이터 — "수동으론 안 됨, 자동화 필수" 확증
- 다자 기능은 향후 팀 합류 시 검증, 또는 가상 시나리오로 보강

## Considered Alternatives

**A) 도그푸딩 없이 빠르게 도구만 만들기**
- 장: 빠른 진행
- 단: 실제 사용 경험 없이 만들면 추상화가 어긋남
- 거부

**B) 부분 도그푸딩 (ADR만)**
- 장: 부담 적음
- 단: 다른 기능들의 검증 누락
- 거부

**C) 전체 도그푸딩 (수동으로라도) ← 선택**
- 장: 모든 기능의 워크플로우 검증
- 단: 수동 부담
- 선택 이유: 도구 개발의 가장 큰 리스크는 "만들어놨는데 안 씀" 이다. 도그푸딩이 이 리스크의 가장 강력한 방어책.
