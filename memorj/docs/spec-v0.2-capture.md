## 1. 판결 — v0.1 §7이 틀린 지점

### 1.1 실측 증거

- **정산 러닝머신**: colin-api 세션(6cd11fa4)에서 Stop 게이트 블록 18회, BLOCK→SETTLE 쌍 12연속. tool_use 문턱(5)은 실질 작업 턴 하나면 넘어서, 게이트가 사실상 매 턴 정산을 강요했다.
- **무한 블록 버그**: settle이 1회라도 있으면 fail-open 상한이 영구 무력화 — 리셋 조건이 "새 settle의 발생"이 아니라 "settle의 존재"였다. 격리 재현 완료 (settle 없는 transcript: 2회 블록 후 통과 / settle 1회 있는 transcript: 무한 블록).
- **몰아쓰기 유실**: 정산 시점 증류는 세션 초반의 발견을 컨텍스트 희석 이후에 복원하게 만든다. 잦은 강제 정산은 억지 기록·빈 정산도 유발했다.
- **동시 세션 충돌**: 상태 파일이 프로젝트 slug 단위라 병행 세션끼리 게이트 카운터를 덮어썼다.

### 1.2 무너진 가정

| v0.1의 가정 | 실측 |
|---|---|
| "왕복 비용: 세션 종료 시 1회 — 무시 가능" (§7.3) | Stop은 **매 턴 종료마다** 발동한다. 세션 종료라는 시점은 하네스에 존재하지 않는다 |
| 정산(턴/세션 끝)이 증류의 시점 | 증류의 최적 시점은 발견의 순간. 발견과 기록 사이의 거리가 유실이다 |
| 강제(블록)가 recall을 만든다 | 강제가 보장할 수 있는 것은 마감뿐. 포착의 recall은 트리거의 명확성에서 온다 |
| 메인 루프만 상정 | 서브에이전트는 SessionStart 주입을 상속받지 않는다(공식 문서 확인) — 채널 없는 사각지대였다 |

### 1.3 살아남은 것 (재확인)

- **§4 프리미티브, §5 상태 기계는 강화됐다.** "포착은 recall에 몰빵, 정밀도는 승격이 회수"가 성립하는 근거가 §5.1(탄생 전부 captured)과 §5.2(활용의 사후 판단)다. 재검토가 이 층을 흔든 게 아니라, 이 층이 새 쓰기 경로를 가능하게 한다.
- §7.1(captured는 raw가 아니라 증류물), §7.2(완전성의 위임 + 포인터)는 유지.
- §7.4(복구 경로)는 유지하되 의미 축소 (→ §7.D).

## 2. 새 §7 — 쓰기 경로: 무엇을, 언제, 어떻게

### 7.A 무엇을 — 기준은 재획득 비용

포착의 판정 질문은 하나다: **"미래 세션이 이걸 다시 지불해야 하는가?"**

- 근거를 갖고 대안을 기각한 결정 → `decision`
- 조사가 도달한 결론, 발견한 제약·환경 사실 → `fact`
- 시행착오를 통과해 얻은 방법 → `learning`
- 사람 차원의 지식(선호·교정)도 같은 기준 → `scope: global`

**가치 판단("흥미로운가", "기록할 가치가 있는가")은 포착 단계에서 하지 않는다** — 그것은 승격(§5.2)의 일이다. 재획득 비용이 0인 것(다시 정해도 비용이 없는 사소한 선택)만 이 기준이 자연 탈락시킨다. type은 이 기준을 통과한 지식의 분류일 뿐, 트리거가 아니다.

이 기준이 사용자용 해설(insight류)과 capture를 가른다: 해설은 precision 지향 큐레이션(적게, 좋은 것만 — 절제가 미덕), capture는 recall 지향 기장(일단 적는다 — 거르기는 관리층이). **방향이 반대이므로 하나를 다른 것의 트리거로 쓰지 않는다.**

### 7.B 언제 — 발견 즉시 (사건 기반)

트리거는 시점(턴 끝·세션 끝)이 아니라 **사건**이다: 결정이 내려진 순간, 조사가 결론에 도달한 순간, 시행착오가 통과한 순간. 그 자리에서 `memory_write`.

1. **유실**: 발견과 기록 사이의 거리가 유실이다. 몰아쓰기는 컨텍스트 희석 후의 복원.
2. **감지 가능성**: "흥미로움"은 재량이라 recall이 낮지만(실측: insight는 가끔만 발동), "방금 결정했다"는 사건이라 놓치기 어렵다.
3. **보편성**: 사건 기준은 세션 전체 맥락 없이도 판정 가능 — 서브에이전트에도 같은 규칙이 성립한다.

### 7.C 어떻게 — 채널 3 + 마감 게이트 1

| 컨텍스트 | 채널 | 성격 |
|---|---|---|
| 메인 (인터랙티브·헤드리스 동일) | SessionStart manifest의 기록 규약 | 지침 |
| 서브에이전트 | **SubagentStart 훅**이 기록 규약(압축판)을 주입 | 구조 |
| 서브에이전트 (보강) | 부모 지침: 위임 프롬프트에 "재획득 비용 있는 발견은 보고에 포함" | 지침 |

- 서브에이전트는 **기록만** 한다. 마감(settle·승격 심판)은 세션 맥락을 가진 메인의 일 — 서브에이전트 규약에서 settle을 언급하지 않는다.
- SubagentStart 채택 근거: 부모의 SessionStart additionalContext는 서브에이전트에 상속되지 않고(공식 문서), CLAUDE.md는 조건부 상속(Explore/Plan 제외). PreToolUse의 updatedInput으로 프롬프트를 수정하는 대안은 가능하나 문서화가 약해 기각.

**마감 게이트 (Stop hook)** — 게이트는 기록을 생성시키지 않는다. **이미 태어난 기록이 닫히는 것만 보장한다.**

- 블록 조건: 마지막 settle 이후 미마감 `memory_write` 존재. 그 외 침묵 — write 없는 세션에게 게이트는 존재하지 않는 것과 같다.
- fail-open: 연속 블록 상한(2회). 카운터 리셋은 settle의 **전진**(transcript 내 settle 횟수 증가)으로만 — "존재"로 리셋하면 무한 블록이 된다 (§1.1의 버그).
- 하네스 채널 제약(실측 확인): Stop에서 에이전트에게 보이는 출력은 block의 reason뿐. 비차단 stdout·systemMessage는 사용자 UI 전용. **"부드러운 알림"은 설계 불가능하다 — 조용함은 곧 침묵이다.** 에이전트용 비차단 통로가 필요하면 UserPromptSubmit이 유일 (v0.2는 사용하지 않음).

**settle의 재정의** — 정산(증류의 의식)에서 **마감(closure)** 으로:

- 역할: 빌려 쓴 captured의 승격 심판 + 작업 단위 닫기 선언. 기록은 이 호출과 무관하게 §7.B에서 이미 일어났다.
- 승격은 in-flow도 허용: 검증을 체감한 순간 promote를 담아 호출해도 된다. 턴 끝 settle은 마지막 기회이지 유일한 기회가 아니다.
- **미마감(未settle)의 정의 축소**: "write 이후 settle 없이 끝난 세션"만. write 없는 세션은 자명하게 닫혀 있다.

### 7.D 복구 경로 (v0.1 §7.4 계승, 의미 축소)

미마감 write를 남긴 채 끝난 세션 → 다음 SessionStart가 감지·경고(세션당 1회, 경고 후 마킹). v0.1과 달리 **지식 유실이 아니다** — 기록은 이미 정본에 있다. 누락된 것은 빌려 쓴 지식의 승격 심판뿐이므로, 복구는 재증류가 아니라 승격 재심 권고다.

## 5. v0.2.1 추가 결정 (2026-07-23)

### 5.1 global 스코프 유예

기록·검색·manifest 모든 경로에서 global(`~/.memory/entries`)을 제외한다. 재도입은 관리 라운드에서 재설계 후.

근거: (1) **오염 반경** — 관리층 없는 recall 지향 포착에서 project 오염은 그 프로젝트에 갇히지만 global 오염은 모든 프로젝트의 모든 세션 manifest에 주입된다. (2) **반쪽 라이프사이클** — global에는 epoch이 없어 휴면 판정이 불가, RECENT도 건수 기반이라 §5 상태 기계가 제대로 돌지 않는 스코프였다. (3) **역할 중복** — 사람 차원 지식은 하네스 자체 메모리(auto-memory)가 이미 담당한다. 부수 효과: 활성 경계가 "`.memory`를 가진(opt-in한) 프로젝트"로 단순해진다 — 이전에는 `~/.memory/entries`만 있으면 아무 디렉토리에서나 훅이 켜졌다. 기존 global 엔트리 파일은 그대로 둔다 (불변식 2 — 시스템은 삭제하지 않는다).

### 5.2 언어 이원화 — 지시문은 영문 고정, 산출물 언어는 설정

에이전트 대면 텍스트(manifest 기록 규약, 훅 메시지, 서브에이전트 주입문, 툴 설명·결과)는 **영문 고정**. 저장 엔트리(title·body)와 사용자 영수증의 언어는 `.memory/config.json`의 `language`(자유 문자열, 기본 `English`)로 프로젝트별 설정.

근거: 지시문은 LLM 준수율·이식성을 위해 영문(커맨드명·지시문 영문 / 산출물 로컬 언어 이원화 원칙과 동일 계보). 저장 언어를 프로젝트별 **고정값**으로 두는 이유는 검색이 grep 기반이라서다 — 저장소에 언어가 섞이면 질의 언어와 다른 엔트리를 영원히 못 찾는다. 언어 일관성이 곧 검색 recall이므로 "대화 언어 따라가기(auto)"는 기각.

### 5.3 기록 영수증 (가시화)

memory_write 직후 에이전트는 사용자에게 압축 영수증 블록(`📝` + `[type] title (id)` + 한 줄 요지)을 표시한다 (manifest 지침, 설정 언어로). 이것은 큐레이션(insight류)이 아니라 기장의 영수증 — 트리거가 "기록 발생" 사건 그 자체라 재량이 없다. 목적: recall 지향 포착의 **사람 쪽 감시 루프** — 무엇이 정본에 쌓이는지 실시간으로 보이면 이상 기록을 입구에서 교정할 수 있다 (불변식 3: 사람의 개입은 항상 이긴다). 서브에이전트는 영수증 대신 최종 보고에 기록 목록을 포함한다.

## 6. v0.3.0 추가 결정 (2026-07-23)

### 6.1 설정 폴백 체인 + `/memorj:setting`

설정 읽기는 프로젝트 `.memorj/config.json` → 사용자 `~/.memorj/config.json` → 내장 기본값 순의 폴백. 언어 같은 사용자 성향을 프로젝트마다 수동 파일로 요구하는 것은 보장 1(캡처에 사람의 규율 0) 위반이라서다 — 사용자 기본값에 한 번 세팅하면 모든 프로젝트에 적용되고, 프로젝트 파일은 "git으로 공유되는 저장소의 언어 고정" 같은 오버라이드 용도로만 쓴다. 설정의 UX는 `/memorj:setting` 슬래시 커맨드 (인자 없이 = 현황 표시 + 초기 세팅, `key value [--project]` = 변경). 키는 확장 가능 — 현재는 language뿐.

### 6.2 저장 루트 리네임 — `.memory` → `.memorj`, MCP 서버명 `memory` → `memorj`

모든 파일 저장(정본·epoch·설정·상태)을 `.memorj`(프로젝트) / `~/.memorj`(사용자) 아래로 옮긴다.

근거는 미학이 아니라 충돌이다: (1) `.memory` 디렉토리는 memorj의 **opt-in 마커**인데, 범용 이름은 다른 도구가 만든 `.memory`에 memorj가 오작동 활성화하는 경로가 된다 — 마커는 소유가 명확해야 한다. (2) MCP 서버명 "memory"는 공식 MCP 예제 서버와 동명 — 가장 충돌하기 쉬운 이름. (3) `.git`·`.claude`처럼 규약 디렉토리는 도구명이 이긴 전례가 있고, 이름이 곧 "누가 이걸 관리하는가"의 답이 된다. 사용자가 1인인 지금이 리네임 비용의 최저점. 기존 프로젝트 마이그레이션은 `mv .memory .memorj` 한 줄 (git 추적 시 `git mv`).
