# Agent Memory Convention — v0.1 Draft (r2)

> 작성일: 2026-07-07 (r2 — 읽기 경로·상태 기계 전면 개정)
> 상태: **의미론 완결.** 모든 개념이 정의됨. 남은 것은 레퍼런스 구현.
> 이 문서는 규약(convention)의 스펙 초안이다. 제품 기획서가 아니다.
> v0.1의 완료 기준: 실체의 완성이 아니라 **개념의 완성** — 모든 의미론이 정의되고, 각각이 최소 구현으로 한 번씩 증명되는 것.

---

## 0. 한 문장 목적

**에이전트 세션에서 생성되는 지식이, 사람의 정리 노력 없이, 축적되어 이후의 모든 세션이 — 도구와 무관하게 — 그 위에서 시작하게 한다.**

### 어절 분해 (각 단어가 배제하는 것)

| 어절 | 의미 | 배제하는 것 |
|---|---|---|
| "에이전트 세션에서 생성되는 지식" | 대상의 한정: 결정·근거·시행착오 | 웹 스크랩, 범용 노트, 문서 정리 |
| "사람의 정리 노력 없이" | 존재 이유. Obsidian 실패의 극복 | 인간 규율을 전제하는 모든 설계 |
| "축적되어" | 풀려는 문제 = 증발 | 세션 단위 리셋 |
| "이후의 **모든** 세션이" | 도구 독립성 | 특정 하네스/벤더 종속 |
| "그 위에서 시작하게" | 환류가 완성 조건 | 저장만 하고 읽히지 않는 시스템 |

### 이것이 아닌 것 (non-goals)

- **팀 동기화가 아니다** — 개인 노드가 core. 팀은 "승격된 지식을 git으로 교환하는" 규약의 미래 확장
- **지능 향상 일반이 아니다** — RAG 최적화, 프롬프트 엔지니어링은 스코프 밖
- **제품이 아니다** — 스펙 + 레퍼런스 구현. 성공 지표는 유저 수가 아니라 "내 삽질이 줄었는가"
- **노트 시스템이 아니다** — 사람의 정리 습관을 요구하는 순간 실패로 정의
- **로깅 규약이 아니다** — raw transcript는 하네스의 소관. 이 규약은 지식 레이어만 소유

## 1. 보장 (Guarantees)

1. **자동 캡처** — 캡처에 사람의 규율이 0이어야 한다. 사람의 역할은 **선택적 개입**뿐 (승격·강등·편집·삭제 — 어느 것도 시스템 작동의 전제가 아님)
2. **사용자 소유 정본** — 정본은 사용자의 디스크에 있고, 어떤 벤더·서비스의 생존에도 의존하지 않는다
3. **보편 가독** — 정본을 읽는 데 특정 런타임, 서버, 계정이 필요 없다
4. **파생물은 재구축 가능** — 인덱스, 캐시, manifest, 동기화 서비스 등 모든 파생물은 정본에서 재구축 가능하며, 정본을 대체할 수 없다

> "파일"은 목적이 아니라 보장 2·3을 현시점에 만족하는 유일한 구현이다.

---

## 2. 불변식 (Invariants)

설계 전체를 관통하는 법칙. 위반하는 제안은 그 자체로 기각 사유.

1. **시간은 어떤 상태도 바꾸지 못한다.** 정본을 만지는 손은 세션 안에만 있다 — 에이전트의 선언 또는 사람의 개입. 배치 프로세스, cron, 데몬은 규약에 존재하지 않는다
2. **정본의 데이터를 파괴하는 손은 사람뿐이다.** 시스템은 어떤 경로로도 지식을 삭제하지 않는다 (휴면은 파괴가 아니다 — §5.4)
3. **사람의 개입은 항상 이긴다.** 에이전트의 어떤 판단도 사람의 명시적 결정을 뒤집을 수 없다
4. **상태 전이의 판단자는 컨텍스트를 가진 자다.** 모든 전이는 그 지식과 접촉 중인 세션(또는 사람)이 트리거한다. 별도의 심사 시점(배치 리뷰, 정리 타임)은 없다
5. **promoted는 시간·불용으로 지위를 잃지 않는다.** manifest를 떠나는 경로는 supersession과 사람의 demote뿐

## 3. 구조 개요

### 정본 (Canonical Form)

- 파일시스템 위의 **plain markdown + YAML frontmatter**
- 근거: 기계(파싱)와 사람(열람) 두 소비자를 동시에 섬기는 유일한 형식

### VCS와의 관계 — 문법은 계승, 기계는 위임

- 정본 저장소는 버전 관리 하에 있는 것이 **권장된다 (SHOULD)**. project 스코프는 repo 안이라 자동 충족, global은 레퍼런스 구현이 `git init`
- **텍스트 수준의 이력은 규약의 소관이 아니며 VCS에 위임된다** — 사람의 promoted 편집의 롤백·감사(blame)·멀티 머신 동기화가 여기서 공짜로 나옴 (위임+포인터 패턴의 세 번째 적용: transcript→하네스 로그, manifest→읽기 시점 생성, 텍스트 이력→VCS)
- 규약은 git의 **문법**(plain text 정본, 재구축 가능한 파생물, 안정적 ref, 히스토리 보존)은 계승하되, git의 **기계**(오브젝트 스토어, 컨텐츠 어드레싱, DAG)는 재구현하지 않는다 — 함께 쓰는 형태이기 때문에 닮으면 중복이고, 사람이 못 읽는 저장소는 보장 3을 죽인다
- 구현 중 판정 기준: **"지금 빌리려는 게 git의 원리인가, 기계인가?"** 원리면 빌리고, 기계면 git에게 시킨다 (예: rename 추적 = 기계 → git의 일 / 충돌하는 결정의 처리 = 원리 → supersession 의미론의 일)

### 소비자 모델

```
                    ┌─ 에이전트 (다음 세션이 읽음)          ← 주입 (§6 읽기 경로)
정본 (파일, 규약) ──┼─ 사람 (Obsidian 등 노트 도구로 열람)   ← 열람
                    ├─ 인덱스 (벡터/SQLite, 재구축 가능)     ← 검색 (v0.1엔 없음, §6.3)
                    └─ 미래의 무언가 (동기화, 대시보드...)
```

- 노트 도구는 읽기 소비자이자, 쓰기는 개입 경로로만 (사람의 파일 편집 = 승격/강등/정정)
- 정리는 **불필요하지만 가능한** 것

### 위치 (잠정)

- `global` 스코프: `~/.memory/`
- `project` 스코프: 프로젝트 루트 `.memory/` (기본) / repo 없으면 `~/.memory/projects/<name>/` 폴백
- ⚠️ 미확정: 디렉토리 내부 구조, 프로젝트 식별자 규칙

## 4. 프리미티브

심문 기준: "이게 빠지면 규약이 성립 안 하는가?"

### 4.1 엔트리 (Entry) — 기록의 원자

- 한 덩어리의 지식: 하나의 결정 / 발견 / 교훈. 턴 요약이 아님
- 물리 단위: **엔트리당 파일 하나**
- 타입 3개 (v0.1에서 더 쪼개지 않음): `decision`(선택+근거) / `learning`(시행착오) / `fact`(발견한 제약·환경 사실)

**품질 기준 — 자기완결성, 두 층:**

| 층 | 기준 |
|---|---|
| **본문** | 원 세션 컨텍스트가 전혀 없는 독자가 그것만 읽고 이해할 수 있다 |
| **제목** | 본문 없이 "무엇에 관한 지식인지 + 당겨볼지"를 판단할 수 있다 — **제목은 발견의 단위** (manifest의 재료, §6.1) |

- ❌ 제목: "커넥터 이슈" / ✅ 제목: "StarRocks 커넥터 대신 raw JDBC (validation-프록시 충돌)"

### 4.2 상태 (State) — 3개, 그 이상 없음

| 상태 | 의미 | 영속성 |
|---|---|---|
| `captured` | 지식 후보. 아직 지속성이 증명되지 않음 | 파일은 영속 (지위는 휴면 가능 — §5.4) |
| `promoted` | 지속성이 증명된 진실. manifest 거주자 | 영속 + 불변식 5 |
| `superseded` | 임기가 끝난 옛 진실. 히스토리 | 영속. 발견 대상에서만 제외 |

- `expired`는 상태가 아니다 — 휴면은 읽기 경로의 성질 (§5.4)
- hot 티어는 규약의 대상이 아니다 — 그것은 컨텍스트 윈도우 그 자체

### 4.3 스코프 (Scope)

| 스코프 | 의미 | 휴면 |
|---|---|---|
| `global` | person 축 — 프로젝트 무관 ("나는 Spark shuffle에서 맨날 이 실수를 한다") | **없음** — 세션 밀도와 무관한 지식, 볼륨 작음, 오탐 비용 큼 |
| `project` | 프로젝트에 속하는 지식 | epoch 기반 (§5.4) |

- branch 스코프 기각 — git은 특정 도구의 개념 (보장 3 위반)

### 4.4 Epoch — 프로젝트의 세션 카운터

- project 스코프별 단조 증가 정수. 작업 단위(세션) 시작마다 +1
- 용도: 휴면 판정의 시계. **벽시계 시간이 아니라 생존 기회의 횟수** — 2주 휴가가 갓난 지식을 잠재우지 않고, 하루 20세션 스프린트에서 느슨해지지 않음 (generational GC의 나이 = 살아남은 collection 횟수와 동형)
- 저장: `.memory/epoch` 파일 (정수 하나). 규약 유일의 가변 공유 상태
- 정밀성 불요: **근사적 단조 증가면 충분** — 용도가 휴면 판정뿐이라 동시 세션의 ±1 race는 무해. lock 없음
- "세션"의 정의는 하네스 어댑터에 위임 (스펙: "구현은 작업 단위의 시작마다 epoch을 증가시킨다")

### 4.5 엔트리 스키마 (frontmatter = memory_write 툴의 input 스키마를 겸함)

```markdown
---
id: 2026-07-07-a3f9
title: StarRocks 커넥터 대신 raw JDBC (validation-프록시 충돌)   # 명시 필드 — 발견의 단위
type: decision            # decision | learning | fact
state: captured           # captured | promoted | superseded
scope: project            # global | project
project: claude-memory    # scope=global이면 없음
epoch: 47                 # 탄생 세션 번호 (project 스코프만)
created: 2026-07-07T14:32:00+09:00   # 사람의 열람·감사용
source: claude-code       # 어느 하네스 (참고용, 의미론 없음)
session: <session-ref>    # 출처 세션 포인터 (완전성 위임, §7.2)
promoted_by: null         # agent | human | supersession — 승격 경로 감사용
supersedes: null          # 대체 선언 (역링크 superseded_by는 파생)
---

풀링 이슈로 커넥터 포기. 재시도 3회 후 raw JDBC + HikariCP로 전환.
근거: 커넥터의 connection validation이 우리 프록시 설정과 충돌...
```

> `memory_write`에 **state 파라미터는 없다** — 탄생 상태는 규약이 정한다 (§5.1)
>
> **필드 최소성 원칙: 필드는 현재 시제의 소비자가 있어야 존재한다** (title→manifest, epoch→휴면 판정, promoted_by→dogfooding 지표). 가정법 소비자("미래에 필요할지도")는 탈락 사유. 포맷 버전 라벨(spec)이 이 원칙으로 기각됨 — frontmatter는 self-describing이라 필드 구성 자체가 세대를 말해주며, git도 오브젝트 원자에 버전을 박지 않는다

## 5. 상태 기계 — 라이프사이클

이 상태 기계가 CLAUDE.md류(정적 파일)에 없는 것 = 이 규약의 차별점의 알맹이.

**레이어 구분**: 이 상태 기계는 **결정의 버전 관리**다 — supersedes는 parent 포인터, superseded 체인은 히스토리, promoted 집합은 HEAD(현재 진실). 반면 **텍스트의 버전 관리**(편집 이력, 롤백, blame)는 규약의 소관이 아니며 VCS에 위임된다 (§3). 같은 파일 위에서 두 레이어가 서로 다른 역사를 관리한다 — "커넥터→JDBC→커넥터v2"는 diff로 복원되지 않는 의미 단위의 역사이므로.

```
                      ┌────────── 사람의 demote ──────────┐
                      ▼                                    │
탄생 ──> captured ── 활용 검증 후 승격 (세션 정산) ──> promoted ── supersession ──> superseded
              │  (또는 사람의 승격, 언제든)                   ▲
              │                                             │
              └── (promoted의 승계자로 탄생 시: 슬롯 상속) ──┘

captured의 휴면: 상태 전이 아님 — 읽기 경로의 성질 (§5.4)
```

### 5.1 탄생 — 전부 captured

**모든 엔트리는 captured로 태어난다.** 쓰기 에이전트는 자기 지식을 승격할 수 없다.

근거 — 증류와 지속성은 다른 판단이다:
- **증류**(무슨 일이 왜 있었나) = 과거의 기술 → 컨텍스트가 자산 → 세션의 에이전트가 최적 판단자
- **지속성**(이게 나중에 중요할 것인가) = 미래의 예측 → 생성 직후의 저자는 **최악으로 보정된 판단자** (방금 내린 결정은 전부 중요해 보임)

저자 승격을 허용하면: 과잉 승격 → manifest 오염 → 사람이 청소 → 정리 노동의 귀환 (보장 1 위반).
**승격은 예언이 아니라 생존이다.** (generational GC: 어떤 객체도 태어나면서 tenured가 아니다. 코드 리뷰: 저자는 자기 PR을 승인하지 못한다)

유일한 예외 — **슬롯 상속** (§5.3): promoted의 승계자는 promoted로 태어난다.

### 5.2 승격 — 활용의 사후 판단

**트리거: 접촉이 아니라 활용 후 판단.** 검색해서 가져온 것은 증거가 아니다 (읽어보니 안 맞았을 수 있음). 지속성의 증거 = 그 지식이 실제 작업을 통과했다는 사실. 판단 시점 = 작업이 끝난 뒤 = **세션 정산 (Stop hook)**.

세션 정산의 이중 역할:
1. 이번 세션이 **낳은** 지식을 기록한다 (전부 captured로)
2. 이번 세션이 **빌려 쓴** 지식을 심판한다 (활용이 검증된 captured를 승격)

- 결정론적 게이트 하나가 두 일을 다 함. 별도 의식 없음 (불변식 4)
- "검색했는데 쓸모없었음" = 정산에서 승격 안 하면 끝. 무액션
- 사람의 승격: 언제든, 어떤 경로로든 (대화 지시, 파일 편집). `promoted_by: human`
- 에이전트 승격: `promoted_by: agent` — dogfooding에서 정밀도 측정 대상 (§9)

### 5.3 Supersession — 정정 (강등이 아니라 교체)

**대체는 새 엔트리의 주장이다.** 새 엔트리가 `supersedes: <옛 id>`를 선언하면 옛 엔트리는 superseded로 전이된다 (결정론적 부수효과, MUST).

- 감지의 재료: 쓰기 에이전트의 컨텍스트에 이미 manifest가 있음 — 옛 진실의 제목을 본 상태로 새 결정을 쓰므로 충돌을 인지할 수 있다
- 감지 실패의 fallback: 사람이 manifest/열람에서 낡은 줄 발견 → 지시 → 에이전트 처리. 완벽한 자동 감지는 개념의 요구가 아님 (메커니즘이 개념, 감지율은 구현 품질)
- 링크의 정본은 새 엔트리의 `supersedes`. 옛 엔트리의 `superseded_by`는 열람 편의용 파생

**슬롯 상속 (원자적 스왑)**: promoted를 supersede하는 엔트리는 promoted로 태어난다 — (옛→superseded, 새→promoted) 두 전이가 한 선언으로. `promoted_by: supersession`.
- §5.1의 예외가 아닌 이유: 저자의 중요도 *예측*이 아니라, **생존으로 이미 검증된 슬롯의 내용 갱신**이므로 (old gen 객체의 in-place 교체이지 nursery 승격이 아님)
- 반쪽만 허용하면(강등만 되고 승계는 안 되면) 구멍: 새 captured가 휴면하면 주제의 지식이 지도에서 통째로 사라짐
- captured를 supersede하는 것은 그냥 captured (상속할 슬롯 없음). 이때 옛 captured는 superseded로 보존하지 않고 자연 휴면에 맡김 — 미검증 후보끼리의 교체는 히스토리 가치가 낮고, 체인 고아 문제를 발생 지점에서 소거

**편집 ≠ 대체**: promoted 파일의 오타 수정·근거 보강은 편집 (사람의 파일). supersession은 결정 *자체*가 뒤집힐 때. 경계의 판정은 사람/에이전트의 몫 — 스펙은 경계의 존재만 명시.

### 5.4 휴면 (Dormancy) — 상태가 아니라 읽기 경로의 성질

**정본에는 아무 일도 일어나지 않는다.** 파일 그대로, frontmatter 그대로. 삭제도, 마킹도, graveyard도, 그것을 실행할 데몬도 없다 (불변식 1·2).

- **잠든다** = 최근 창(§6.1)에서 밀려나고 검색 랭킹에서 후순위가 됨 — 읽기 시점에 계산되는 성질
- **판정** = `현재 epoch − 탄생 epoch > K` (K는 구현 상수). 의미: **생존 기회가 K번 주어졌는데 활용되지 않음**
- **깨어난다** = 깊은 검색이 닿음 → 활용 → 정산에서 승격. 별도 부활 메커니즘이 아니라 §5.2 그대로
- decay의 정체 (최종): 저장 레이어의 파괴가 아니라 **읽기 레이어의 망각**. 뇌가 기억을 지우는 게 아니라 인출 강도가 떨어지는 것. 신호/소음 분리는 에이전트의 *시야*에서 달성되면 충분하고, 정본은 조용히 전부 보유
- 귀결: "중요한데 안 닿은 지식"의 실패 모드가 **소실 → 찾기 어려움**으로 강등. 후자는 검색 품질(파생 레이어)의 개선 대상이지 규약의 상처가 아님
- v0.1은 최후 접촉 추적 안 함 (읽기가 쓰기를 유발하면 정본이 오염). 접근 통계는 파생 인덱스의 자유

### 5.5 하강 계열 용어 정리

| 용어 | 무엇 | 판정 대상 | 트리거 | 후계자 | 남는 것 |
|---|---|---|---|---|---|
| **휴면** (dormancy) | captured가 시야에서 밀림 | 가치 (미증명) | 시간 아닌 epoch, 읽기 시점 계산 | 없음 | 전부 (파일 무변) |
| **대체** (supersession) | promoted의 정정 | 정확성 (사실 아니게 됨) | 선언 | **필수** | superseded + 체인 |
| **강등** (demote) | promoted→captured | 지위 (승격이 과했음) | **사람만** | 없음 | captured로 복귀 |

## 6. 읽기 경로 (주입) — 관련성 문제

주입은 공짜가 아니다: 컨텍스트 윈도우가 진짜 비용이고, 무관한 주입은 0이 아니라 마이너스(컨텍스트 오염). 쓰기가 recall이면 읽기는 precision.

### 읽기 의미론 (MUST/MAY)

> - promoted는 세션 시작 시 **발견 가능**해야 한다 (MUST — 존재와 정체, 제목 수준)
> - promoted는 요청 시 **조회 가능**해야 한다 (MUST)
> - captured는 **검색 가능**해야 한다 (MUST)
> - 최근 captured는 세션 시작 시 발견 가능해야 한다 (MUST — §5의 탄생 룰 하에서 신생 지식의 유일한 생존 기회 창구이므로, 성립 조건)
> - superseded는 발견 대상에서 제외된다 (MUST)
> - 전문 주입 여부·형태는 구현의 선택

### 6.1 Manifest — 세션 시작 시 주입되는 지도

**파일이 아니다 — 읽기 시점에 정본에서 생성되는 파생물** (보장 4). SessionStart hook이 frontmatter+제목을 스캔해 즉석 조립. 쓰기 경로는 manifest를 모른다.

2부 구성, 모두 제목 수준 (엔트리당 ~15토큰):

```
[PROMOTED — 이 스코프의 진실]
[decision] StarRocks 커넥터 대신 raw JDBC (validation-프록시 충돌)  (id: a3f9)
[learning] Glue에서 pyspark.sql.functions 와일드카드 임포트 금지     (id: b2e1)
...전체...

[RECENT — 최근 K epoch의 미검증 후보]
[fact] 스테이징 DB는 매주 월요일 리셋됨                              (id: c4d2)
...최근 창만...
```

- 설계 원리: **"모르는 걸 모르는" 상태만 제거하고, 아는 비용은 on-demand로 미룸.** 에이전트는 자기가 모르는 걸 검색할 수 없다 — 가드레일 지식("이 repo는 JDBC")의 가치는 문제의식이 없는 순간에 끼어드는 데 있고, pull은 "질문이 생기기 전에 알아야 하는 지식"에 구조적으로 실패한다
- manifest가 아파지는 스케일(수천 promoted) = 주입 방식의 문제가 아니라 승격 기준이 무너졌다는 신호

### 6.2 Pull — 툴 2개

- `memory_search(query)` — captured 포함 전체 검색. **v0.1 구현은 grep 기반** (정본이 plain text이므로 가능)
- `memory_read(id)` — manifest에서 본 제목의 본문 조회

### 6.3 벡터 인덱스 — v0.1에 없음

- 근거: 개념의 완성이 목적. 검색 *인터페이스*는 개념(스펙에 남음), 벡터는 실체(파생물). dogfooding 스케일에서 grep 대비 이득은 미검증 가정
- grep이 아파지는 시점에 붙이는 첫 파생물. 인터페이스 불변이므로 교체 무비용

### 6.4 런타임 한 장 요약

```
세션 시작:  epoch +1 → manifest 주입 (promoted 전체 + 최근 captured, 제목만)
작업 중:    필요시 pull (search/read)
            작업이 기존 지식과 닿으면 → 그 자리에서 상태 조정
            (supersession 선언 — 작업의 부수효과)
세션 정산:  낳은 지식 기록 (전부 captured) + 빌려 쓴 지식 심판 (검증된 것 승격)
사람:       언제든 개입, 항상 승리
배경:       아무것도 없음 — 데몬 없는 시스템 (불변식 1)
```

별도의 "메모리 관리 시간"이 어디에도 없다는 것 — 이것이 보장 1의 존재 증명.

## 7. 쓰기 경로 (캡처) — 완전성 문제

### 7.1 captured는 raw가 아니라 증류물

1. 원 제약: "I/O는 하네스가 이미 저장. 저장하고 싶은 것은 생각의 과정과 논리"
2. 읽기 경로가 강제: 저장 시점에 증류 안 된 것은 읽기 시점의 빚
3. 목적문의 주어가 "지식"

### 7.2 완전성의 위임 + 포인터

> 규약은 transcript를 저장하지 않는다. 모든 엔트리는 출처 세션 참조(`session:`)를 가진다.
> 완전성은 하네스의 세션 로그가 담당하고, 규약은 지식 레이어만 소유한다.

- 증류 누락 = 소실이 아니라 지연 (세션 로그에서 재증류 가능)

### 7.3 증류 채널: tool call + Stop hook 게이트

thinking 블록이 추론의 전용 출력 채널이듯, 지식 증류에도 전용 채널이 필요하다. 오늘의 구현 = tool call.

structured output의 보장을 분해해 plugin 레벨에서 재조립:

| 보장 | 메커니즘 |
|---|---|
| 출력이 스키마에 맞는다 | `memory_write` MCP 툴 — JSON 스키마 검증. **state 파라미터 없음** (탄생 상태는 규약이 결정), **title 필수** (제목의 자기완결성) |
| 출력이 반드시 일어난다 | **Stop hook 게이트** — 차단 권한으로 발생을 강제 |

```
턴 종료 시도 → Stop hook (결정론적) → memory_write 호출 확인
  → 없으면 block + "정산하라: 이번 세션의 지식 기록 + 빌려 쓴 지식 심판"
  → 정산 → 재종료 → 통과
```

- 스킵이 불가능한 게 아니라 스킵이 결정론적으로 적발·교정되는 구조
- 왕복 비용: 세션 종료 시 1회 — 무시 가능
- 품질 논거: 증류의 최적 판단자는 그 컨텍스트를 살고 있는 에이전트 자신 (지속성 판단과 분리 — §5.1)
- 스펙 표현: 규약은 "세션 종료 전 증류 채널을 통한 정산"이라는 의미론을 정의하고, 채널의 구현은 하네스별 (Claude Code = MCP tool + Stop hook)

### 7.4 엣지 케이스

1. **무한 루프 가드**: 기록할 게 없는 세션을 위한 빈 정산 선언(`no_entries: true`). hook은 발생만 검사, 개수를 강요하지 않음
2. **비정상 종료**: Stop hook 미발동 → 다음 SessionStart가 "직전 세션 미정산" 감지 → 세션 로그 기반 후처리. 후처리는 주 경로가 아니라 **예외 복구 경로**

## 8. 차별점 (복기용)

### vs LLM 공급자 native memory
보장 1은 그들이 더 잘함. 2·3·4는 **구조적으로** 불충족 — 벤더 메모리의 존재 이유가 락인이므로 충족은 자기 부정. 관계 설정: 경쟁이 아니라 git↔GitHub — 벤더가 이 규약의 파일을 읽고 쓰는 소비자가 될 수 있는 구조.

### vs 서비스향 memory 도구 (Mem0, Zep, Letta...)
정본이 그들의 DB/API (보장 2·3 불충족). 고객이 다름 — B2D 인프라, 카파시 질문의 주어(개인)가 그들 시장엔 없음. 기능으로 경쟁하지 않는다: 축을 바꾼다 — 서버의 DB가 아니라 내 디스크의 파일, 앱의 기능이 아니라 앱들이 읽는 규약.

### 한 줄 압축
> 둘 다 메모리를 자산으로 만들어 자기 쪽에 두는데, 이 규약은 메모리를 사용자 쪽에 두고 자신은 형식으로 남는다.

### 정직한 리스크
1. **진짜 경쟁자는 벤더발 파일 규약** (CLAUDE.md, agents.md) — 승부처는 파일 기반이 아니라 **라이프사이클 의미론** (§5 상태 기계가 알맹이)
2. **절실한 인구는 소수** — 초기 사용자상 = 도구를 갈아타고 데이터 주권에 예민한 개발자 니치. 결함 아님: 규약은 절실한 소수에서 시작해 도구가 대신 퍼뜨림
