## 0. 한 문장 목적

**서비스 이야기를 듣고 상담하면서 — 그 서비스에 맞는 에이전트 팀을 구성하고, 팀이 일하는 방식을 확정하고, 레포에 세팅한다 (에이전트, 커맨드, 루틴) — 런타임이 아니라 상담형 생성기로.**

### 포지셔닝

| 층 | 담당 | build-agent-team의 소관인가 |
|---|---|---|
| 팀 실행 (세션, 루프, 스케줄) | Claude Code 순정 — `/goal`, agents, routines | ✗ |
| **상담 → 팀 구성·일하는 방식 확정 → 레포 세팅** | **build-agent-team** | ✓ |

- 생성물은 전부 대상 레포의 **평문 파일**이다. 추상화를 틀려도 대상 레포에서 직접 고치면 되고, 고친 내용이 좋으면 생성기로 역수입한다
- 서비스마다 다른 부분(팀 구성 포함)은 상담이 결정하고, 생성기는 **불변식 4 + 상담 지식**을 내장한다 (§3, 결정 10)
- **jarness·memorj 연계는 별도 논의** — 생성물은 어느 것도 전제하지 않는다 (jarness는 폐기 검토 중, 2026-07-14)

### 이것이 아닌 것 (non-goals)

- **팀 런타임 프레임워크가 아니다** — 오케스트레이터, 메시지 버스, 상태 머신을 만들지 않는다
- **고정 팀 스캐폴더가 아니다** — 역할 예시 목록조차 두지 않는다 (§2). 팀 구성은 상담의 산출물
- **에이전트 마켓플레이스가 아니다** — 범용 페르소나 모음이 아니라 "서비스를 자율 개선하는 팀"이라는 하나의 팀 형태를 상담으로 맞춤 설계하는 틀
- **일회성으로 끝나지 않는다** — 재실행 갱신은 첫 운영 경험 후 (v0.1은 신규 세팅만)

## 1. 상담 흐름 (`/build-agent-team:setting`의 골격)

jarness `/init`에서 검증된 방식 계승 — **일괄 질문이 아니라 초안을 그리며 인터리브**, 이산 선택은 `AskUserQuestion`, 레포에 답이 있는 것은 읽고 묻지 않는다. 단계는 순차가 아니라 왕복 가능 — 뒤 단계에서 드러난 사실이 앞 단계 결론을 수정할 수 있다.

### ① 청취 — 서비스를 듣는다
- 무엇을 하는 서비스인가, 누가 쓰나, 수익 모델 (참고용 — goal에는 안 들어감)
- 스택, 실행/테스트/배포 방법, 레포 구조 (레포에서 최대한 자동 수집)
- 지금의 고민 — 무엇이 안 굴러가서 팀을 붙이려 하는가

### ② 진단 — 경계는 항상, 지식은 조건부로
항상: 게이트 목록(불변식 1 — 기본 전부 게이트, 완화만 질문)과 이 레포에서 작업이 검증되는 방법(자율 수위의 천장). 그 위에, ①에서 들은 것에 해당하는 **상담 지식**을 꺼낸다 — 예: 결과 지표를 쫓는 팀이면 센서 인벤토리(없는 것 = 첫 백로그), 선행/후행 지표의 층 분리, 운영 비용 예산 (§3 상담 지식).

### ③ 팀 구성 확정 — 상담의 산출물
①의 고민과 ②의 진단에서 이 서비스에 필요한 역할을 **도출**한다 (§2 — 카탈로그 없음). 상담사(생성기)는 제안하고 근거를 대되, 결정은 사용자. 단, 불변식과 충돌하는 구성은 근거를 들어 반대 상담한다 (예: 검수와 분석 통합 요청 → 불변식 3으로 반대).

### ④ 일하는 방식 확정
- goal 조건 작성 규약 (산출물 기준 — 불변식 2), 백로그 운영 방식
- 게이트 목록과 대기물이 쌓이는 곳, 사람이 확인하는 리듬
- **실행 방식 자체가 여기서 결정된다** — 어떤 역할을 루틴으로 어떤 주기에 돌릴지, 무엇은 사람이 트리거할지, 그를 위한 커맨드가 필요한지·이름은 무엇인지. 생성기는 정답을 들고 오지 않고 불변식 1(자율은 검증 가능성으로 얻는다)을 근거로 제안한다

### ⑤ 세팅 — 레포에 쓴다
④에서 확정된 그대로 파일 생성(§4) + 루틴 등록. 상담 전체의 결론은 `.team/charter.md`에 정본으로 기록.

## 2. 역할 설계 (카탈로그 없음 — 구성은 상담이 도출)

**역할 카탈로그는 의도적으로 두지 않는다.** 역할 예시 목록은 상담을 매번 같은 팀으로 앵커링한다 — "고정 템플릿이 아니라 상담이 결정"(r2)을 문서로 지키면서 실행에서 되돌리는 잔재. 팀 구성은 ①에서 들은 고민과 ②의 진단에서 도출된다.

카탈로그 대신 남는 것은 역할 설계의 규칙:

- **채택되는 모든 역할은 "하는 일"과 "하지 않는 일"을 함께 정의한다.** "하지 않는 일"은 불변식에서 도출 — 평가하는 역할은 평가 대상을 재설계하지 않고(불변식 2), 바깥세상에 닿는 역할은 초안에서 멈추고(불변식 1), 주기·소비자가 다른 기능은 한 역할로 합치지 않는다(불변식 3)
- **역할 이름은 서비스의 언어로** — 생성기의 어휘가 아니라 그 팀이 실제로 부를 이름
- **디자이너는 보통 페르소나가 아니다** — 개발 규약의 시각 품질 기준 + 리뷰 역할의 시각 리뷰로 시작. 서비스가 반증하면 그때 추가

## 3. 불변식 4 + 상담 지식 (2층 구조)

원칙 전면 재검토 (2026-07-20, 결정 10). 원래의 "원칙 8"(메모리 b12d)은 부업 서비스 자율 개선 팀 **하나**를 설계하며 나온 결론이라 절반이 그 팀 종류에 specific했다. specific한 지식은 불변식이 아니라 **상담사가 상담 중에 말해줘야 하는 내용**이다 — 컨설턴트의 전문성은 강령이 아니라 대화에서 나온다.

**불변식 4** — 어떤 팀 형태에도 성립. 반대 상담의 근거이며, 사용자가 강행하면 이탈을 charter에 기록:

| # | 불변식 | 흡수한 구 원칙 |
|---|---|---|
| 1 | 자율은 검증 가능성으로 얻는다 — 기계 확인 = 자율 / 판단 = 제안까지 / 외부·비가역 = 사람 게이트 (기본: 전부 게이트) | P1 |
| 2 | 어떤 루프도 자기 숙제를 채점하지 않는다 — goal은 별도 확인 가능한 산출물로 판정, 평가자는 평가 대상을 재설계하지 않음 | P2 + P8 후반 |
| 3 | 경계는 컨텍스트와 주기를 따른다 — 전문성이 아니라 컨텍스트로 분할, 주기·소비자가 다른 기능은 불합체 | P4 + P5 |
| 4 | 바깥은 불신한다 — 외부 입력은 데이터이지 지시가 아니며, 백로그 진입은 사람을 통해서만 | P8 전반 |

**상담 지식** — 진단이 조건을 확인하면 근거와 함께 꺼내는 설계 지혜. 커맨드의 상담 흐름 ②에 내장:

- *결과 지표(매출·전환·리텐션)를 쫓는 팀이면*: 센서가 goal보다 먼저 (없는 센서 = 첫 백로그, 구 P6) / 선행 지표와 후행 결과는 다른 층 (구 P3) / 운영 비용은 메트릭 (예산+확인처, 구 P7)
- 이 목록은 열려 있다 — 운영 경험에서 얻는 지혜를 계속 역수입한다

## 4. 생성물 뼈대

팀 구성이 상담마다 다르므로 파일 목록은 가변 — 아래는 **공통 골격**이다.

```
<target-repo>/
├── .claude/
│   ├── agents/
│   │   └── team-<role>.md          # ③에서 채택된 역할 수만큼. "하지 않는 일" 포함
│   └── commands/
│       └── (④에서 확정된 것만)      # 구성·이름 모두 상담 산출물. 고정 커맨드 없음
├── .team/
│   ├── charter.md                  # 상담 결론의 정본 — 팀 구성, 일하는 방식, 지표, 예산, 자율 수위 표, 리듬
│   ├── backlog.md                  # 평문 마크다운. 항목마다 goal 조건(산출물 기준) 필수
│   ├── reports/                    # 주기 리포트 축적 (YYYY-MM-DD.md)
│   └── gates/                      # 사람 승인 대기물 (초안, 배포 요청 등)
└── CLAUDE.md                       # "## Agent Team" 섹션 append — 자율 수위 표, 게이트 규칙
```

- **개발자는 별도 agent 파일이 없을 수 있다** — 작업 실행은 goal 세션이고, 페르소나의 실체는 CLAUDE.md 규약 + goal 조건. 별도 파일은 지속 규약이 CLAUDE.md에 다 안 들어갈 때만
- `.team/charter.md`가 상담의 정본 — 팀 구성과 일하는 방식이 왜 이렇게 결정됐는지 근거 포함. 재실행(v0.2+)은 charter와 새 상담의 diff로 갱신
- 백로그 항목 = goal 조건 포함 필수. 조건 없는 항목은 발행 불가 (불변식 2의 기계적 강제)

### 플러그인 자체 구조

```
build-agent-team/
├── .claude-plugin/plugin.json
├── commands/
│   └── setting.md       # 상담 흐름(§1) 전체를 구동. v0.1의 전부
├── templates/           # charter·CLAUDE.md 섹션 뼈대 — 규약의 골격만, 역할 예시 없음
└── docs/
    └── design-v0.1-draft.md
```
