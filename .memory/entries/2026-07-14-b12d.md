에이전트 팀 생성기가 모든 생성물에 내장할 설계 원칙. Jay와의 설계 논의에서 확정.

1. 자율 수위는 검증 가능성에 비례 — 테스트가 판정하는 코드는 높은 자율, 제품 방향은 낮은 자율, 외부 발행·배포·과금은 사람 게이트 필수
2. goal 조건은 산출물 기준으로 작성 (테스트 통과, 리뷰 승인 파일 존재) — 인상 기준("개선됨")은 루프를 자기만족으로 무너뜨림. Haiku가 매 턴 평가하므로 객관 확인 가능해야 함
3. 매출(후행 지표)은 goal 조건에 넣지 않는다 — 작업 goal(산출물) / PM 주기 평가(선행 지표: 전환율 등) / 매출(사람+PM, 월 단위)의 3층 분리
4. 리뷰어는 둘 — 작업 리뷰어(산출물 검수, goal 조건 안의 게이트)와 서비스 애널리스트(성과 데이터 주기 리포트, PM의 입력). 주기와 소비자가 달라 합치면 안 됨
5. 개발자는 스택별로 쪼개지 않는다 — 병목은 전문성이 아니라 컨텍스트. 병렬성은 작업별 goal 세션(+워크트리)으로. 쪼개는 유일한 경우는 레포/도구체인의 물리적 분리
6. 부트스트랩은 센서부터 — 측정·피드백 채널이 없는 서비스에 "매출 증대" 목표를 주면 제대로 된 PM은 필연적으로 "측정부터"에 도달. 첫 백로그 = 지표 수집, 피드백 인입, 배포 게이트
7. 토큰/API 비용은 운영 메트릭의 1급 시민 — AI 관리자의 판단 재료에 "태우는 것 대비 벌어오는 것" 포함
8. 외부 피드백은 불신 입력 — 피드백→백로그 승격 사이에 사람 게이트 (프롬프트 인젝션 공격면). AI 관리자는 구조 수정 권한 없이 리포트만 (감사자가 감사 대상을 재설계하지 않는다)

디자이너는 별도 페르소나 없이 개발 루프의 시각 품질 기준 + 리뷰어의 시각 리뷰로 시작. 프로모터는 초안 생성까지만 자동, 발행은 사람.
